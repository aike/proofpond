# 問題追加スキル

ユーザーが示したざっくりした問題のアイデアから、ProofPond の問題データ一式を生成して登録する。

## 前提知識(必ず最初に読む)

1. `src/content/schema.ts` — 問題・知識のZodスキーマ(必須フィールドの正)
2. `src/content/README.md` — コンテンツ執筆規約(ヒントの粒度、模範解答・最終ゴール解説の書き方)
3. 既存の問題1つ(例: `src/content/problems/injective-kernel.json`)— フォーマットとトーンの見本
4. `src/content/knowledge/*.json` — 既存の知識エントリ(再利用のため全idを把握する)

## 手順

### 1. 問題文の清書

ユーザーの入力を大学初年次〜中級レベルの線形代数の証明問題として適切な文章に清書する。

- 記号・量化を厳密に(「〜とする」「〜を示せ」の形)
- 数式は `$...$` / `$$...$$`、JSON内ではバックスラッシュを二重に(`\\mathbb{R}`)
- 曖昧さがある場合(例: 一般の $n$ 個か2個か、$\mathbb{R}$ 上か一般の体か)は、初学者向けに簡単な方を選び、選択した旨を最後の報告で伝える。ユーザーの意図が本当に判別できない場合のみ質問する

### 2. 設計

- **id**: 内容を表す英語のケバブケース(例: `image-subspace`)。既存の問題idと重複しないこと
- **difficulty**: 1(定義の確認で解ける)/ 2(定石の組み合わせ)/ 3(発想の一段階が必要)
- **knowledgeIds**: 証明に使う定義・定理・性質・考え方を列挙。**既存エントリを最優先で再利用**し、足りないものだけ新規作成する
- **中間ゴール**: 3〜6個。「定義を書く」「仮定を整理する」「〜を示す」など学習者の手が動く単位。最後のゴールで証明が完結すること

### 3. 不足している知識エントリの追加

必要な知識が `src/content/knowledge/` に無ければ作成する。

- 分野が合う既存ファイル(`vector-space.json` / `linear-map.json` / `eigen.json` など)に追記。合うファイルが無ければ新しい分野ファイルを作る
- 全フィールドを埋める: `id`, `name`, `kind`(definition/theorem/property/technique/example), `statement`, `relatedIds`(相互リンクを既存側にも追加検討), `usage`, `tags`

### 4. 問題JSONの生成

`src/content/problems/<id>.json` に以下の構造で書く(スキーマが正。ここは要点のみ):

```jsonc
{
  "id": "...",
  "title": "...",              // 一覧用の短い題
  "statement": "...",          // 清書した問題文
  "difficulty": 1,
  "knowledgeIds": ["..."],
  "finalGoal": {
    "statement": "...",        // 示すべき命題への翻訳
    "why": "...",              // なぜこれを示せば問題の証明になるのか(定義による言い換えの十分性を明示)
    "howToFind": "..."         // 思いつくために必要な知識と発想方法(他の問題でも再利用できる形で)
  },
  "subgoals": [
    {
      "id": "...",
      "title": "...",
      "description": "...",    // 任意
      "hints": [ { "level": 1, "text": "..." } ],  // レベルは1から連続
      "solution": "..."        // 必須。このステップの清書された証明文
    }
  ],
  "overallHints": [ { "level": 1, "text": "..." } ],  // 全体の見取り図(1〜2個)
  "tags": ["..."]
}
```

生成時の品質基準(執筆規約の要約):

- **ヒントの粒度**: Lv1=方向性のみ / Lv2=使う定義・定理の名指し / Lv3=証明の流れ・式変形の途中まで / Lv4=ほぼ完成形(清書はしない)。全中間ゴールに最低Lv1〜2、詰まりやすいゴールにはLv3〜4まで
- **模範解答**: そのステップの範囲だけを清書する。先のステップの内容を先取りしない。最終ステップは $\blacksquare$ で締める
- **finalGoal.why**: 「これを示して本当に証明になるのか?」に答える。たいてい「定義によりこの言い換えと元の主張は同じ」という論理の説明になる
- **finalGoal.howToFind**: 参照すべき知識名を挙げ、「示すべき語の定義に立ち返る」のような再利用可能な発想手順として書く
- 数学的な正しさを最優先。生成後、証明の流れを頭から通して検算すること(仮定の使い漏れ・論理の飛躍がないか)

### 5. 検証

```
npm run validate
```

エラー(参照切れ・ヒントレベル不連続・必須フィールド欠落など)が出たら修正して再実行。通るまで終わらない。

### 6. 報告

ユーザーに以下を簡潔に報告する:

- 清書した問題文(選択した解釈があればその旨)
- 追加したファイル(問題JSON、新規知識エントリ)
- 中間ゴールの構成(タイトル一覧)
- `npm run dev` で `#/problems/<id>` を開けば確認できること
