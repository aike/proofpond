# コンテンツ執筆規約

問題・知識エントリを追加するときのルール。`npm run validate` で機械検証されるもの以外に、以下を守ること。

## ファイル配置

- 知識: `knowledge/<分野>.json` — 1ファイルに複数エントリの配列
- 問題: `problems/<問題id>.json` — 1ファイル1問題

## LaTeX の書き方

- インライン数式は `$...$`、別行立ては `$$...$$`
- JSON文字列内なのでバックスラッシュは二重にする(`\\mathbb{R}`、`\\mathbf{v}`)
- 改行は `\n`(画面では `white-space: pre-wrap` でそのまま改行される)
- KaTeX でレンダリングされる。使える記法は https://katex.org/docs/supported.html

## ヒントの粒度(最重要)

ヒントは各レベルで次の粒度を守る。清書された解答文は `solution`(模範解答)にのみ書き、ヒントには書かない。

| レベル | 粒度 | 例 |
|---|---|---|
| 1 | 方向性のみ。何に注目するか | 「W の一般の元を文字でおくところから始める」 |
| 2 | 使うべき定義・定理・知識カードの名指し | 「u, v とおくと W の元であることから何が言える?」 |
| 3 | 証明の流れ。式変形の途中まで | 「仮定は x₁+y₁+z₁=0 と…。和の成分の総和を計算する」 |
| 4 | ほぼ完成形。ただし清書された解答文にはしない | 計算の完成形+「…と結んで証明完了」 |

- レベル4でも「証明の完全な清書」は書かない。清書は模範解答(`solution`)の役割
- レベルは1から連続で付ける(1→2→3→4。途中を飛ばさない)
- すべての中間ゴールに最低レベル1〜2のヒントを付ける。レベル3〜4は詰まりやすいゴールにのみ付けてもよい

## 模範解答(solution)

- **全中間ゴールに必須**(スキーマで強制)。そのステップの答案として通用する、清書された証明文を書く
- そのステップの範囲だけを書く。先のステップの内容を先取りしない(UIでは現在・達成済みのゴールしか開示されないが、内容面でもネタバレを避ける)
- 最後のステップの模範解答は証明全体の締め($\blacksquare$ など)まで書く

## 最終ゴール(finalGoal)

3項目とも必須。

- `statement`: 示すべき命題。問題文の主張を検証可能な形に翻訳したもの
- `why`: **なぜこれを示せば問題の証明になるのか**。多くの場合「定義によりこの言い換えと元の主張は同じ」という論理の説明になる。学習者が「これを示して本当に証明になるのか?」と不安にならないよう、十分性を明確に述べる
- `howToFind`: **この最終ゴールを思いつくために必要な知識と発想方法**。参照すべき定義・定石の名前を挙げ、「示すべき語の定義に立ち返る」「⇔は2方向に分ける」のような再利用可能な発想の手順として書く。この問題に限らず他の問題でも使える形で言語化すること

## 中間ゴールの設計

- 「定義を書く」「仮定を書く」など、学習者の手が動く単位で分ける(仕様4.3)
- 1問あたり3〜6個が目安
- `description` は補足があるときのみ

## 知識エントリ

- `relatedIds` と `usage`(利用場面)を必ず書く。相互リンクがUIの知識カード導線になる
- `kind` は definition / theorem / property / technique(考え方) / example(典型問題)
